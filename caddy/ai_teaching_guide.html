🧠 心法：如何馴服 AI 進行高效 pair-programming

寫程式不是拼速度，而是精準度。與 AI 共舞有三大黃金定律：

定律一

「先討論，別動手」的決策緩衝

AI 的天性是「討好人類」，收到需求會立即開始寫程式碼，甚至會大範圍覆寫您的專案。在交付複雜任務前，先下達指令限制其修改權限，要求它「先列出修改計劃並詢問回饋，待確認後才可動手」，這是防範專案被 AI 寫爛的最強手段。

定律二

架構切片與局部實作

不要一次叫 AI 寫完一整套系統。以本專案為例，我們先叫 AI 「只下載 Caddy 並啟動」，再叫它「寫 Python API 讀取 JSON」，然後是「設計前端面板」，最後才是「串接 Cloudflare」。一步一步驗證，Bug 就不會滾雪球。

定律三

自建 AI 讀取手冊 (Manifest)

當專案變大、對話被截斷時，AI 會忘記上下文。每次開發到一個段落，讓 AI 寫一個專為 AI 解析的 PROJECT_SUMMARY.md。當新的 AI 接手時，叫它第一步先讀這份檔案，即可瞬間無縫融入專案。

🚀 FALO Go Gateway 實作心路歷程

回顧我們是如何一步步完成這套高度自動化的雲端轉址網關系統：

階段一：建立獨立閘道結構與下載 Caddy

建立獨立的 caddy/ 目錄以利後續 Git 對應。AI 撰寫 Shell 腳本自動辨識 macOS 架構（Intel vs Apple Silicon），精準下載官方 Caddy 二進位檔。

階段二：架設 Python API 與 Flask 後台

使用 Flask 3.1 實作輕量級管理伺服器。定義 rules.json 結構化儲存，並在此階段將原訂 Port 8010 更改為 8088，成功排除本地衝突。

階段三：介面優化與 macOS launchd 常駐自啟

實作 Glassmorphism 毛玻璃暗黑風管理面板。優化轉址路徑點擊跳轉與「一鍵複製公網網址」按鈕，並編寫 macOS launchd plist，實現開機免開終端機自動背景運作。

階段四：Cloudflare API 自動對接

這也是最酷的自動化部分！我們整合專案的 SDK，撰寫腳本自動註冊 CNAME DNS 紀錄，並寫入 Tunnel 的遠端 Ingress 路由，徹底排除公網訪問的 404 故障，真正做到全自動化部署。

階段五：安全防護與訪問統計

因應公網暴露需求，為後台加入 Basic Auth（帳密 falo / force）。同時讓 Caddy 輸出 JSON 格式日誌，後端解析生成實時連線統計，支援欄位排序、搜尋、CSV 匯出及清除紀錄。

🛡️ AI 開發防踩坑守則 (防 Bug 聖經)

在開發這套網關的過程中，我們解決了數個底層硬體與服務溝通的經典 Bug，未來的開發者與 AI 應引以為戒：

Bug 1：Python 啟動背景 Daemon 被 PIPE 阻塞 hang 死

現象：當使用 Python 的 subprocess.Popen 啟動 Caddy 或 Tunnel 時，如果將 stdout/stderr 設定為 subprocess.PIPE，Python 會陷入無限期阻塞等待。
防踩坑解法：因為背景進程會一直運作，Pipe 的緩衝區滿了就會卡死。應該將輸出重定向至實體日誌檔案（如 open('caddy.log', 'w')）或使用 DEVNULL，不要用 PIPE！

Bug 2：Caddy 全域日誌與站台存取日誌 (Access Log) 區別

現象：在 Caddyfile 全域宣告了 `log` block 後，發現 `/test` 的存取紀錄依然沒有寫入 `caddy.log`。
防踩坑解法：Caddy 的全域 `log` 只記錄 Caddy 系統本身的啟動和錯誤。若要記錄網站的訪問日誌（用於解析連線紀錄），必須在網站區塊 (Site Block) 內部單獨再配置 `log` 指令。

Bug 3：macOS 系統 DNS 快取造成的 ERR_NAME_NOT_RESOLVED

現象：在 Cloudflare 註冊完 CNAME 後，Mac 用 `nslookup` 可以成功解析 IP，但瀏覽器和 `ping` 卻一直報 `ERR_NAME_NOT_RESOLVED` 錯誤。
防踩坑解法：這是 macOS 系統服務 `mDNSResponder` 快取了「未配置 DNS 時的失敗結果」（負向快取）。應手動在 Mac 終端機執行 sudo killall -HUP mDNSResponder 清除快取，並打開 Chrome 無痕視窗測試。

📋 精選實用 Prompt 範本

您可以直接拷貝這些 Prompt 來訓練或命令其他 AI 助手：

範本 A. 啟動專案前（鎖定「先討論」原則）：

我需要你在專案目錄下進行一項新功能的開發。但在你讀取完目錄檔案後，請遵守「先討論，不動手」的原則：
1. 請先為我分析現有架構，並列出您計畫修改或新增哪些程式碼檔案。
2. 列出可能產生的風險（如埠口衝突、進程阻塞）。
3. 在我回覆「同意修改」之前，請不要對程式碼做任何實際的修改或寫入！

範本 B. 指令生成 AI 讀取清冊 (Manifest)：

請幫我整理目前為止我們所有修改過的程式檔案、系統連接埠、控制流邏輯以及踩坑點。
請將這些內容輸出為一個名為 PROJECT_SUMMARY.md 的 Markdown 檔案，以方便未來的 AI 助手讀取後能立即融入開發。請確保內容結構清晰，並在結尾署名 Falo x Force Cheng 2026/6/28。
